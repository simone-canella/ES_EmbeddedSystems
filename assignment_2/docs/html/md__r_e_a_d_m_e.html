dsPIC33 Buggy Control Firmware Documentation

This repository contains the embedded control firmware for the autonomous buggy project. The architecture is engineered around a dual \( O(1) \) deterministic Finite State Machine (FSM) implemented on a Microchip dsPIC33 microcontroller. It provides predictable, real-time behavior without using conditional switch-cases or blocking delays.

System Architecture

To ensure strict real-time execution safety, the system separates State Transitions (triggered by spontaneous external events) from State Actions (continuous operations running at fixed frequencies) using two flat functional pointer tables with \( O(1) \) computational complexity.

1. Dual \( O(1) \) Optimization Mechanics

  • The Transition Matrix (transition_table): A 2D array of function pointers indexed directly by [Current_State][Incoming_Event]. It updates the state machine variables instantaneously upon event arrival.
  • The Cyclic Action Array (cyclic_action_table): A flat array of function pointers indexed directly by [Current_State]. The main scheduler executes the behavior of the current active state via a single calculated jump (call) instruction, eliminating variable-time switch-case evaluation paths.

Finite State Machine Specification

The system processes three distinct functional operational states:

1. STATE_HALTED

  • Entry / Behavior: The robot starts initialization in this state.
  • Motor Actuation: The PWM Duty Cycle (DC) of all four H-Bridge control signals (RD1–RD4) is set strictly to 0%.
  • Indicator Configuration: The left-side (RB8) and right-side (RF1) lights flash synchronously at a frequency of 1 Hz. Low-intensity auxiliary lights (RG1) are forced OFF.
  • Transitions: Pressing the physical button RE8 switches the system to STATE_MOVING.

2. STATE_MOVING

  • Motor Actuation: The 10 kHz motor PWM duty cycles on pins RD1–RD4 are adjusted dynamically inside a 500 Hz control loop matching the current speed and yaw rate references received from the UART interface:

    \[ left\_pwm = speed - yawrate \]

    \[ right\_pwm = speed + yawrate \]

    Saturation limits are locked strictly between \( -100\% \) and \( +100\% \).
  • Indicator Configuration: Left and right-side lights are turned completely OFF. Low-intensity auxiliary lights are turned ON.
  • Transitions: * Pressing button RE8 drops the system back into STATE_HALTED.
    • If the 500 Hz IR sensor subsystem (AN14/AN15) senses an obstacle within the critical threshold range, the firmware immediately moves into STATE_OBSTACLE_AVOIDANCE.

3. STATE_OBSTACLE_AVOIDANCE

  • Behavior Maneuver: The robot halts forward instruction and executes an automated, non-blocking time-tracked hardware maneuver:
    1. Rotate 90 degrees clockwise.
    2. Advance forward continuously for exactly 2.0 seconds.
    3. Rotate 90 degrees counter-clockwise to realign with the initial bearing angle.
  • Indicator Configuration: The right-side lights blink at 1 Hz, while left-side lights stay dark and low-intensity lights remain active.
  • Evaluation Loop Retry Logic: After the 2-second directional maneuver completes, the system samples the IR sensor again.
    • If the path is now clear, it returns to STATE_MOVING.
    • If the obstacle remains, the recovery sequence repeats for a maximum limit of 3 attempts. If the path remains obstructed after the third attempt, the system enters the safe STATE_HALTED fallback state.
  • Transitions: Pressing button RE8 at any point instantly stops the sequence and shifts the machine to STATE_HALTED.

Firmware Integrity & Safe Integration

  1. Non-Blocking Execution Rules: Software timing loops must never utilize blocking delay functions (e.g., __delay_ms()). Timing tracks for structural state tasks (such as the 2-second avoidance maneuver) must evaluate increments computed within the periodic 500 Hz Timer Interrupt.
  2. Interrupt Data Guardrails: Any telemetry arrays or state structures shared between the main execution thread and the UART/Timer Peripheral Interrupt Service Routines (ISRs) must be declared with the volatile qualifier.
  3. Atomic Operations: To ensure total data integrity, wrap data exchanges or state mutations inside brief critical section blocks using assembly built-ins (__builtin_disi(0x3FFF)) to temporarily disable nested interrupts during shared buffer modifications.
